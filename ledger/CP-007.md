### Auth layer

**`frontend/src/lib/auth/`**
- `types.ts` — `AuthUser`, `AuthSession` (with `issuedAt`), `AuthContextValue`.
- `storage.ts` — read/write/clear helpers for the `cp.auth.v1` localStorage key. Guards against SSR and storage errors.
- `provider.tsx` — `AuthProvider` client component + `useAuth` / `useInitials` hooks. Exposes `signIn`, `signUp`, `signOut`, `updateUser`. Hydrates from storage after mount so initial SSR matches `session=null`.
- Placeholder semantics: any non-empty email + password works; `signUp` requires at least a 6-char password. Display name is derived from the email's local-part if not provided.

### Routing

Reorganized into a route group so the auth shell and the app shell don't fight:

```
app/
├── layout.tsx                   # root: html + AuthProvider
├── (protected)/
│   ├── layout.tsx               # auth guard + AppShell
│   ├── page.tsx                 # dashboard
│   ├── scanner/…
│   ├── symbol/[ticker]/…
│   ├── alerts/…
│   ├── backtesting/…
│   └── settings/…
├── login/page.tsx               # standalone, no AppShell
├── register/page.tsx            # standalone, no AppShell
└── not-found.tsx                # standalone
```

- `(protected)/layout.tsx` is a client component. It reads `useAuth()`, shows a spinner while loading, and `router.replace("/login?next=<path>")` if unauthenticated. The `next=` param round-trips back to the originally requested page after sign-in.
- All six existing pages moved with `git mv` so history is preserved.
- `/login` and `/register` wrap their `useSearchParams()` call in `<Suspense>` to satisfy Next's static-prerender contract.
- `not-found.tsx` updated to own its own full-viewport chrome (no longer assumes AppShell around it).

### Top bar

Replaced the decorative avatar button with a real shadcn `DropdownMenu`:
- Trigger shows the signed-in user's initials (derived in `useInitials`).
- Content: user name + email, a Settings link, and a Sign out action.
- Sign out clears the session and `router.replace("/login")`.

### Profile section

New `components/settings/ProfileSection.tsx` (client):
- Initializes Name + Email from the current session via `useEffect`.
- Save button is disabled until the form is dirty; on submit calls `updateUser` and shows a transient "Saved." status for 1.5s.
- Persists through `AuthProvider`, which writes back to localStorage.

The settings page imports `<ProfileSection />` where the inline form used to be. Every other settings section is unchanged.

## Decisions & notes
- **Placeholder, not Clerk/Supabase.** Brief explicitly said "placeholder auth provider initially." Swap path: drop the provider SDK's `AuthProvider` in over mine, replace `signIn`/`signUp`/`signOut`/`updateUser` with SDK calls. `useAuth` consumers (TopBar, ProfileSection, the protected layout) don't change.
- **Client-side route guard.** With localStorage-only session there's no server-side cookie to gate on via middleware. The guard is in `(protected)/layout.tsx`. Real auth will replace it with Next middleware reading the SDK's session cookie.
- **`?next=` preserved.** A user who lands on `/settings` while signed out goes to `/login?next=%2Fsettings` and returns to Settings on success.
- **Didn't persist other settings toggles yet.** Notifications/Theme/Risk still use uncontrolled inputs. The CP-006 pattern (`useAlertsStore`) is a clean template when those need to persist — intentionally deferred so this sprint ships tight.
- **TopBar dropdown works signed out too** (shows just Settings + Sign out), but the guard means the user won't reach a page with the top bar while signed out. Left the conditional in place defensively.

## Not in scope (deferred)
