# CP-009 — Pattern Engine

**Date:** 2026-04-19
**Goal:** Detect Hammer, Doji, Engulfing, Shooting Star, and Inside Bar on closed candles. Return pattern, direction, confidence, and timestamp.

## Delivered

### New package `backend/app/patterns/`

```
patterns/
├── types.py       # DetectedPattern, PatternName, PatternDirection
├── detectors.py   # 5 pure functions + ALL_DETECTORS tuple
└── engine.py      # PatternEngine — per-(symbol,tf) rolling buffer + history
```

### Types
- `PatternName = Literal["hammer", "doji", "engulfing", "shooting_star", "inside_bar"]`
- `PatternDirection = Literal["bullish", "bearish", "neutral"]`
- `DetectedPattern(symbol, timeframe, pattern, direction, confidence, time)` with `confidence: int = Field(ge=0, le=100)` and `time` in Unix seconds (the close-time of the bar the pattern confirmed on).

### Detectors (`detectors.py`)
All pure functions over `Sequence[Candle]` (most recent last), returning `(pattern, direction, confidence) | None`. Scoring scheme: **base 40** for matching the structural criteria + up to **60 shape points** for how cleanly the candle matches (shadow-to-body ratios, compression, engulfing size).

| Detector            | Criteria                                                                                       | Direction          |
| ------------------- | ---------------------------------------------------------------------------------------------- | ------------------ |
| `detect_hammer`       | lower shadow ≥ 2× body, upper shadow ≤ body, body in top third of range                        | bullish            |
| `detect_shooting_star`| mirror of hammer — upper shadow ≥ 2× body, lower shadow ≤ body, body in bottom third            | bearish            |
| `detect_doji`         | body ≤ 10% of total range                                                                       | neutral            |
| `detect_engulfing`    | 2-bar: prev body fully covered by curr body, curr flips colour                                  | bullish / bearish  |
| `detect_inside_bar`   | 2-bar: curr.high < prev.high **and** curr.low > prev.low                                        | neutral            |

### Engine (`engine.py`)
- Per-`(symbol, timeframe)` rolling buffer of up to 20 closed candles (more than enough for the 1-2 bar patterns; room to spare for future multi-bar patterns).
- `on_closed_candle(symbol, tf, candle) → list[DetectedPattern]` pushes the bar, runs every detector, records matches in a per-key detection history (capped at 200 for the REST endpoint), returns the matches.
- `recent(symbol, tf, limit)` returns the tail of the detection history.
- Multiple detectors can co-fire on the same bar (a tight inside bar is often also a doji). All matches are kept.

### Service integration (`market_data/service.py`)
- `MarketDataService.__init__` now accepts `pattern_engine: PatternEngine | None`.
- In the main loop, after publishing each candle event, `update.closed` triggers `engine.on_closed_candle(...)` and every returned match is published to the **same** `StreamHub` with `type: "pattern.detected"`. Clients subscribed to `(symbol, timeframe)` receive both candle and pattern events on one WebSocket.

### API
- `GET /api/v1/patterns/{symbol}?timeframe=1h&limit=50` → `PatternsResponse { symbol, timeframe, patterns: DetectedPattern[] }`. Validates the timeframe; returns 503 if the engine is not enabled.
- WS event shape:
  ```json
  {
    "type": "pattern.detected",
    "symbol": "AAPL",
    "timeframe": "1h",
    "pattern": {
      "symbol": "AAPL", "timeframe": "1h",
      "pattern": "hammer", "direction": "bullish",
      "confidence": 80, "time": 1776646020
    }
  }
  ```
- App `lifespan` constructs `MarketDataService(_build_provider(), pattern_engine=PatternEngine())`.

## Verification

**Detector unit checks** — hand-fed textbook candles:
```
hammer         bullish  80
shooting_star  bearish  93
doji           neutral  87
engulfing      bullish  64
engulfing      bearish  90
inside_bar     neutral  70
```
Each rejection path also verified (oversized upper shadow rejects hammer; current bar extending prev high rejects inside bar; oversized body rejects doji).

**Engine walk** — a 7-bar seeded sequence produced 9 detections across all 5 patterns; `recent()` returned them in insertion order.

**Live backend** —
- `GET /api/v1/patterns/AAPL?timeframe=1h` → 200 with `{"symbol":"AAPL","timeframe":"1h","patterns":[]}` before any bars have closed. Shape correct.
- 75-second WS session subscribing to `NVDA/1m`: 150 `candle.update`, 1 `candle.closed`, 0 `pattern.detected`. The single closed random-walk bar did not match any of the five textbook patterns — correct behavior. The engine ran on it; none fired.

## Decisions & notes

- **Same hub, separate event type.** Candle and pattern events share `(symbol, timeframe)` subscriptions. Clients that care about one get both; those that only care about candles just ignore `pattern.detected`. Simpler than a parallel hub and matches what the frontend actually consumes per symbol.
- **Multiple patterns per bar.** Every detector runs on every closed bar. A doji that's also an inside bar produces two `DetectedPattern`s with distinct confidences. Preserved rather than collapsed because downstream (alerts, strategy engine) may care about specific patterns.
- **Confidence is interpretable, not calibrated.** Scores are a textbook-fit measure, not a backtested win-rate probability. A 90-confidence hammer means "cleanly shaped hammer", not "90% win rate." Real probability calibration is a CP-010/CP-011 concern.
- **Closed-only detection.** Patterns run strictly on `update.closed` events — running on in-progress bars would emit and retract the same pattern many times per bar.
- **Pure detectors, stateful engine.** Detectors never reach into buffers they don't own; the engine owns rolling state. Makes each detector trivially unit-testable.

## Not in scope (deferred)

- Multi-bar patterns beyond 2-bar: Morning/Evening Star, Three White Soldiers/Black Crows. Adding them is: new detector function + append to `ALL_DETECTORS`.
- Trend context. A hammer "after a downtrend" is a bullish reversal; a hammer in an uptrend is just noise. We're not factoring the surrounding trend into confidence yet — a CP-013 (AI) or CP-010 (strategy engine) concern.
- Persisting detections to Postgres — history is in-process only. Restart loses it.
- Frontend consumption of live `pattern.detected` events (chart markers, alert card). Backend is ready; wiring this into the UI is a focused mini-sprint.
