# CP-006 — Alerts Center

**Date:** 2026-04-19
**Goal:** Alerts page with New / Triggered / Saved Rules tabs, notification settings, a Create Alert modal, and mock persistence.

## Delivered

### Audit of existing work
Alerts UI was built during the Figma port — tabs, cards, rules, and a sticky side-panel creation form. CP-006 had three real gaps:

| Brief item                | Status before | Gap                                                       |
| ------------------------- | ------------- | --------------------------------------------------------- |
| New alerts                | ✅            | no id; trash button was decorative                        |
| Triggered alerts          | ✅            | same                                                      |
| Saved rules               | ✅            | switch / edit / delete were decorative                    |
| Notification settings     | ✅            | unchanged                                                 |
| **Create alert modal**    | ❌            | was a sticky side-panel only; no modal                     |
| **Mock persistence**      | ❌            | nothing actually persisted between renders                 |

### New in this sprint

**`frontend/src/lib/mock-alerts.ts`** — types + mock data with proper `id` fields:
- `AlertEvent` (shape used by both New and Triggered tabs)
- `SavedRule`
- `AlertRuleDraft` (form payload; the store assigns the id)
- `mockNewAlerts`, `mockTriggeredAlerts`, `mockSavedRules`

**`frontend/src/hooks/use-alerts-store.ts`** — client-only store:
- Actions: `createRule`, `toggleRule`, `deleteRule`, `dismissNewAlert`, `deleteTriggeredAlert`, `resetToMocks`.
- Hydrates from `localStorage` after mount under key `cp.alerts.v1`. Initial SSR uses the mock defaults so there's no hydration mismatch.
- Silent fall-through on storage errors (quota / blocked): state still lives in memory.
- Generates ids via `crypto.randomUUID()` with a fallback for older environments.

**`frontend/src/components/alerts/AlertRuleForm.tsx`** — extracted form:
- Pure form with controlled state for symbol, signal type, target price, min confidence, and three notification channels.
- Takes an `onSubmit(draft)` callback; form resets after submit.
- `idPrefix` prop so two instances (side panel + dialog) can coexist on one page without colliding input ids.

**`frontend/src/components/alerts/CreateAlertDialog.tsx`** — new:
- Wraps `AlertRuleForm` in a shadcn `Dialog` with Cancel + Create actions.
- Controlled `open` state closes the dialog when a rule is created.
- Default trigger is a "+ Create Alert" button; caller can override via `trigger` prop.

**`CreateAlertPanel.tsx`** — reduced to a thin wrapper around `AlertRuleForm` + the sticky card chrome. Kept because the Figma design shows it; it now shares the same code path as the dialog.

**`AlertCard` / `SavedAlertRule`** — now accept callbacks:
- `AlertCard` takes `onDismiss(id)`; the trash button fires it. The arrow button is now a real link to `/symbol/{slug}` (stripping the `/` from symbols like `BTC/USDT`).
- `SavedAlertRule` takes `onToggle(id)` and `onDelete(id)`; the switch is now controlled by `isActive` instead of `defaultChecked`.

**Alerts page (`app/alerts/page.tsx`)** — became a client component using `useAlertsStore`. Header gained a "+ Create Alert" button that launches `CreateAlertDialog`. Every tab renders a typed empty-state when its list is empty.

## Verification
- `pnpm build` passes. Alerts route: 12.3 → 24.8 kB (Dialog component + controlled form + store hook; still static since the page is prerenderable).
- Build generates all 8 routes including the dynamic `/symbol/[ticker]`.
- Create → Toggle → Delete rule flow, and Dismiss on new/triggered alerts, all persist across reloads via `localStorage`.

## Decisions & notes
- **Kept the side panel alongside the modal.** Brief says modal; Figma shows side panel. Serving both: side panel on `lg+` (always-visible, Figma fidelity), dialog button in the header (brief requirement, also the mobile entry point). Both submit through the same store action.
- **localStorage, not backend.** "Mock persistence" interpreted literally. The store hook's public surface (`createRule(draft)`, etc.) is what the backend integration in a later sprint will replace; callers won't change.
- **Dismiss = remove.** "New Alert" dismiss removes the row outright rather than moving it to Triggered. Simpler, matches what users do with notifications anywhere else. Reconsider when CP-008 streams real events.
- **Edit buttons remain decorative.** Edit-in-place requires opening the modal pre-filled and routing updates through a distinct `updateRule` action. Left for a focused sprint since the brief didn't ask for it.

## Not in scope (deferred)
- Editing an existing rule (the Edit pencil icons are no-ops).
- Notification delivery (email / push / SMS) — CP-008+ on the backend.
- Triggered-alerts stream from the scanner — CP-008.
- Per-rule notification channel editing UI — the rule form has the switches but there's no way to change channels on an existing rule.
- Syncing notification-preferences switches to the store (they're currently uncontrolled in `NotificationPreferences.tsx`).
